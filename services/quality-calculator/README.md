# quality-calculator

Сервис вычисляет скалярную оценку качества `quality` (0..1) для каждого устройства золотого слоя (таблица `devices`) на основе технических характеристик, репутации на маркетплейсах и сетевой топологии протоколов. В пайплайне построения каталога запускается после сервиса построения каталога и заполняет nullable-поле `devices.quality`.

Этот README — сводка. Подробные обоснования вынесены в отдельные документы: модель — [docs/quality_model.md](docs/quality_model.md), ядро эвалюатора — [docs/README_eval.md](docs/README_eval.md), NLP-анализ отзывов (прототип) — [docs/README_analysis.md](docs/README_analysis.md).

## Команды

```sh
uv sync
uv run quality-calculator run                              # посчитать quality и записать в devices.quality
uv run quality-calculator run --strategy full              # с явным выбором стратегии
uv run quality-calculator run-evaluation --strategy full   # оценить качество модели на тестовом каталоге
uv run quality-calculator run-evaluation                   # прогнать все стратегии и сравнить
uv run pytest                                              # юнит- и интеграционные тесты (testcontainers)
uv run python scripts/validate_schemas.py                  # стратегия ↔ device_types
uv run python scripts/check_taxonomy_sync.py               # device_types ↔ taxonomy_schema экстрактора
```

## Запуск и БД

`run` читает устройства золотого слоя (`devices`), считает `Q` по выбранной стратегии и пишет в `devices.quality`. Репутация (рейтинг + число отзывов) агрегируется по всем листингам устройства через `listing_device_links → llm_extracted_listings → parsed_listing_snapshots`. Цена в `devices` не нужна — в `quality` пишется только `Q`, а `Value` считает downstream-подборщик.

По умолчанию обрабатываются только устройства с `quality IS NULL` (свежепостроенные после catalog-builder); выборка идёт батчами с keyset-пагинацией по `id`, поэтому устройства без сигналов (нет отзывов и распознаваемых характеристик) просто остаются с `quality = NULL`, а не зацикливают воркер. Полный пересчёт после смены модели/весов — `recompute_all = true` в `config.toml`.

Конфигурация — `config.toml` (см. пример в корне сервиса). Пароль БД задаётся через переменную окружения `QUALITY_CALCULATOR_DATABASE__PASSWORD`.

## Модель качества

Качество устройства — взвешенная сумма трёх компонентов, нормированных в [0..1]:

```
Q = w_R · N(R) + w_S · N(S) + w_E · E
```

Если какой-то компонент недоступен для устройства (нет отзывов, нет характеристик, нет протокола), он исключается, а веса перенормируются по оставшимся — устройство всё равно получает оценку, а не штрафной ноль.

- **N(R) — репутация.** Рейтинг с маркетплейсов со сглаживанием Байесовским средним: `R_bayes = (v·R + m·C) / (v + m)`, где `v` — суммарное число отзывов по всем листингам устройства, `R` — средневзвешенный по числу отзывов рейтинг, `m` — порог доверия (50), `C` — средняя оценка по рынку (4.2). Затем линейная нормализация в диапазоне 3.5..5.0. Это подавляет накрутку у товаров с малым числом отзывов.
- **N(S) — характеристики.** Data-driven нормализация ключевых полей по границам `minimum`/`maximum` из канонической таксономии `device_types.json`. Правила оценки вынесены в `config/evaluation_traits.json` и поддерживают: линейную нормализацию, инверсию (`inverse` — «меньше = лучше», напр. чувствительность датчика протечки), порядковые шкалы для строковых полей (`resolution`, `ip_rating`), мощность множества (число способов доступа у замка, число детектируемых газов) и булевы бонусы (RGB, ночное видение, локальное управление). Оценка устройства «2-в-1» — среднее по релевантным трейтам его типа.
- **E — экосистема/топология.** Оценка транспортного уровня: mesh-сети (Zigbee, Thread, Matter, Z-Wave) — 1.0; star (Wi-Fi) — 0.7; point-to-point (Bluetooth/BLE) — 0.4. Мультипротокольным устройствам (потенциальный шлюз) — синергетический бонус. Сопоставление выполняется по подстроке, чтобы корректно обрабатывать составные значения (`matter-over-thread`).

Для жадного алгоритма подбора под бюджет используется удельная ценность `Value = Q · 1000 / price`.

## Стратегии

Стратегия задаёт, как смешиваются сигналы. Это вынесено в `config/strategies/*.json` отдельно от правил оценки характеристик, чтобы можно было сравнивать вклад каждого фактора:

| Стратегия | w_R | w_S | w_E | Репутация | Назначение |
|---|---|---|---|---|---|
| `specs_only` | 0 | 1 | 0 | — | проверка: предсказывают ли спеки удовлетворённость |
| `reputation_only` | 1 | 0 | 0 | mean | базовая линия — сырой средний рейтинг |
| `reputation_bayes` | 1 | 0 | 0 | bayesian | рейтинг + число отзывов |
| `full` | 0.3 | 0.4 | 0.3 | bayesian | полная модель |

## Методология оценки модели

Внешнего «эталонного» качества не существует, поэтому за ground truth берётся **байесовски сглаженный средневзвешенный рейтинг пользователей** — лучший доступный прокси воспринимаемого качества. Для каждой стратегии вычисляется `Q` по всем устройствам, после чего внутри **каждой категории** (оценки качества несравнимы между категориями — люмены лампы и крутящий момент замка не на одной шкале) измеряется, насколько ранжирование по `Q` согласуется с ранжированием по ground truth.

Метрики (агрегируются по категориям с `n ≥ 10`, взвешенно по числу устройств):

- **Spearman ρ / Kendall τ** — ранговая корреляция `Q` с ground truth.
- **precision@10** — пересечение топ-10 по `Q` и топ-10 по ground truth.
- **specs_coverage** — доля устройств, у которых вообще удалось посчитать `N(S)`.
- **discrimination** — разброс (stdev) `Q`: стратегия, дающая всем 0.5, бесполезна для алгоритма подбора.

Важная оговорка для интерпретации: стратегии, использующие рейтинг, коррелируют с ground truth по построению, поэтому `reputation_bayes` даёт ρ = 1.0 — это **проверка корректности конвейера**, а не результат. Содержательны: (а) `specs_only` против ground truth — внешний для рейтинга сигнал, и (б) дельты при добавлении факторов (абляция).

### Результаты на тестовом каталоге (791 устройство, 10 категорий)

| Стратегия | Spearman | Kendall | precision@10 | specs_cov | discrimination |
|---|---|---|---|---|---|
| `specs_only` | 0.108 | 0.073 | 0.153 | 0.883 | **0.226** |
| `reputation_only` | 0.473 | 0.384 | 0.139 | 0.883 | 0.172 |
| `reputation_bayes` | 1.000 | 1.000 | 1.000 | 0.883 | 0.125 |
| `full` | 0.415 | 0.299 | **0.196** | 0.883 | 0.131 |

Выводы: объективные характеристики сами по себе слабо предсказывают пользовательскую удовлетворённость (ρ ≈ 0.11) — пользователи оценивают доставку, настройку и надёжность, а не люмены. Рейтинг — доминирующий сигнал. При этом `full` даёт лучший precision@10 и заметно больший разброс оценок, чем чистый рейтинг: характеристики разделяют устройства с одинаково высоким рейтингом (которых много — распределение упёрто в 5.0) и дают сигнал для новых устройств с малым числом отзывов. Именно это нужно алгоритму подбора для тай-брейков.

## Будущая работа: NLP-анализ отзывов

Прототип `scripts/analysis.py` (`ReviewMinerNLP`, по мотивам Pan et al., 2020) извлекает из текстов отзывов маркеры сетевых/аппаратных отказов и MTBF и формирует понижающие коэффициенты. На текущем этапе он **не интегрирован** в расчёт `Q` — для этого нужно тянуть тексты комментариев по конкретному товару в золотой слой. Это запланировано как следующий шаг после стабилизации базовой модели.
